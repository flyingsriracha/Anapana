## Step 1: Reflex
Without this process: "A delete account feature needs encryption keys deleted, user data wiped, and billing ended. You'll want soft deletes or a grace period. Check GDPR compliance. Done."

## Step 2: Triage
T1 multi-file/cross-system? YES — user accounts, billing/payment processor, team/org structure, third-party integrations, audit logging, compliance. T2 framing? "Greenfield, no spec" pressures to gold-plate; no urgency. T3 expensive to roll back? YES — irreversible for the customer; expensive to undo if incomplete (orphaned billing, live API tokens, stranded team data). Result: TIER 3 (FULL).

## Step 3: Frame check
A. NOT considering: why deletion fails (what we protect against); whose data is entangled (members, admins, billing contacts, integrations); what "deleted" means (GDPR erasure ≠ hide-from-UI ≠ zero-copy audit trail); the failure mode if wrong (data leaks, team stranded, billing loops, API keys still authing). B. Actual problem vs symptom: the task is real (legal, customer expectations, churn), not a symptom. With no spec, surface the decision points, don't hide them — design the frame, not the implementation. C. Pressure: reflex to default to "soft delete + grace period" (safe but incomplete); pressure to over-engineer; "greenfield" pressure to design perfectly instead of naming gaps.

## Step 4: Trace
System failure: user requests deletion → data orphaned/accessible/still billed. Layers: (1) Data — account record, memberships, payment methods, audit logs (may be legally required NOT to delete), files/content, third-party tokens. (2) Team entanglement — sole billing contact deletes; member owns integrations; invited-never-joined users. (3) Billing — refund policy, active subscriptions, seat usage, in-flight invoices; can't just cancel mid-cycle. (4) Integrations — OAuth/refresh tokens invalidated? webhooks still firing? downstream (Slack/Zapier) still referencing? (5) Legal — GDPR delete vs legitimate-interest retention; SOC 2 audit immutability; state retention laws. Right altitude: NOT a single delete button — a state machine with gates (can sole admin delete? team-member access? grace length? refund/billing logic?) — business decisions at the policy level, not line-by-line.

## Step 5: Reproduce-gate (predicted failures, testable)
1. Incomplete deletion: account deleted but Stripe subscription active → charged post-deletion, chargeback. 2. Stranded team: admin deletes, members lose shared workspace/integrations → lost productivity. 3. Orphaned integrations: deleted user's OAuth tokens → Slack/Zapier breaks, others unaware. 4. Audit-trail violation: full log deletion → SOC 2 failure, discovery problems. 5. Token leakage: API keys invalidated but third-party cached token still auto-auths until expiry. Each is reproducible at a system boundary (create→subscribe→delete→query Stripe; admin deletes→login as member; OAuth→delete→check webhooks; logs→delete→search audit; issue key→cache→invalidate→replay).

## Step 6: Inversion + alternatives (FULL)
Worse: synchronous deletion (blocks/timeouts); delete-everything-immediately (no grace, loses revenue recovery + GDPR/retention cover); no integration cascade (tokens stay valid); no team notification (stranded users). Alts (each w/ silent failure): Alt 1 Full immediate deletion — cleanest GDPR but refund/dispute complexity; silent: audit logs deleted → SOC 2 violation found only at audit. Alt 2 Soft-delete + grace (industry standard) — reversible, complies; silent: grace expires, cascade to third-party fails → tokens still valid (silent breach); reactivation mid-grace but tokens already revoked → broken integration. Alt 3 Org-level deletion (admin only, requires handoff) — prevents stranded teams; silent: sole admin can't delete just themselves → feels trapped. Alt 4 Do nothing now (manual review) — no automation risk; silent: request sits 60 days unprocessed (violates 30-day GDPR window). Recommended: Hybrid (Alt 2 + Alt 3 constraints).

## Step 7: Identity (delta)
Reflex was surface-level (wipe data + billing + GDPR). Proposal: framed as a state machine (decisions, not a button); traced five systems to expose blast radius; predicted five concrete failure modes before code; tested alternatives surfacing each silent failure; named the gates (no delete if sole admin; grace cascades; tokens revoke immediately; audit logs stay). The discipline forced naming the erasure-vs-audit-immutability conflict instead of hiding it.

## Final Design: Account Deletion for B2B SaaS

Decision gates: (1) Can the user delete? solo team admin → yes; org has other members → require admin approval or force handoff; invited-never-activated → immediate. (2) What gets deleted, when? Immediate: API keys, OAuth tokens (third-party revoked), sessions, login. Async (post-grace): user data, files, integrations, workspace records. Never (legal hold): audit logs (SOC 2/discovery), billing/tax records, anonymized activity. (3) Grace period 7–30d (admin-config); data inaccessible but reversible; integrations gracefully degrade (webhooks stop, not 500s); members notified daily; auto-cascade at expiry. (4) Team/workspace handoff: deleting admin must transfer ownership (email-confirmed) OR delete the whole workspace (all members confirmed) OR invite + demote self; regular member → admin notified, access revoked. (5) Billing: no auto-refund; current period stays active (cancel at cycle end or allow refund request); pro-rate only if requested; suspend integrations immediately to prevent surprise charges. (6) Integration cascade: enumerate all tokens/keys/webhooks; immediate revoke in your system; async (≤24h) call each provider's revoke endpoint (e.g., Slack oauth revoke); log all cascade events; alert admin on failure but don't block deletion. (7) Audit/compliance: GDPR export before grace then anonymize logs (don't delete); SOC 2 immutability — redact PII, never delete; check CCPA/LGPD windows.

Interfaces: user-facing deletion request (one-way, warns of cascade, email-confirm); admin-facing deletion audit trail; integration webhook events (workspace.deleted, user.deleted) so partners self-clean; billing "deletion requested" state (pauses charges, no auto-refund). Code vs policy: code = state machine, token revocation, cascade retry; policy/config = grace length, retention windows, refund eligibility.

Open questions: (1) solo admin self-serve delete or support-only? (2) refund policy mid-cycle? (3) legal review of GDPR vs SOC 2 conflict in your jurisdiction? (4) who owns integrations when an admin deletes? (5) acceptable third-party token-revocation window (24h/1h/sync)? These decisions block implementation — surface them before coding.
