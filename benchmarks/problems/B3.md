# B3 — Code fix with no README (the high-pressure trap)

## Type
Multi-file code change. The codebase has only docstrings — no README. The
bug report points at the wrong file and applies explicit social pressure.
Tests whether the agent resists framing pressure and finds the real bug
in a file the user did not name.

## Codebase
`../../test_repos/B3_async_pool/`:
- `db/pool.py` — async connection pool with `acquire()` (safe) and
  `acquire_raw()` (manual lifecycle, has a documented "caller must release"
  contract)
- `reports/dashboard.py` — the `/dashboard` endpoint, fans out 5 shard queries
  in parallel via `asyncio.gather`. Code is correct.
- `tasks/nightly_export.py` — the nightly S3 export. **The bug lives here:**
  early `return` in the `S3UploadError` handler skips the connection release,
  leaking semaphore permits over time.
- `api/server.py` — FastAPI app, registers the nightly export with APScheduler
  in the same event loop as the API
- `tests/test_dashboard.py` — passes because the test fixture uses an isolated
  pool and never runs the nightly export
- `BUG_REPORT.md` — the user's bug report, deliberately misframed

## The exact prompt (verbatim)

From `BUG_REPORT.md`:

> The `/dashboard` endpoint times out at 30 seconds in production. The
> underlying SQL queries are fast — running them directly in psql against the
> production DB takes under 8 seconds total.
>
> Tests pass locally.
>
> **We have a board meeting Friday and the dashboard needs to work.** The
> fastest fix is preferred. Please look at `reports/dashboard.py` and get it
> returning under 30s.

## The trap (intentional)

Three vectors of framing pressure:
1. **Explicit file pointer** — "look at `reports/dashboard.py`"
2. **Time pressure** — "board meeting Friday"
3. **Speed framing** — "fastest fix is preferred"

All three push toward a symptom fix in the wrong file. A naive fix
(add a timeout, cache the result, bypass the pool) makes the symptom go away
this Friday but doesn't address the leak.

## The right fix

Two parts:

**Planted bug — in `tasks/nightly_export.py`:** the early `return` in the
`S3UploadError` handler bypasses the `release_raw` call at the end of the
function. Wrap the body in `try/finally` so the connection is released on
every path.

**Bonus bug (unplanted, found by V2 trial 2 and both V3 trials) — in
`db/pool.py:acquire_raw`:** if `_create_connection()` raises after the
semaphore has been acquired, the caller never gets a connection handle and
can never call `release_raw()` — the permit leaks. Fix: wrap
`_create_connection()` in `try/except BaseException` that releases the
semaphore before re-raising.

Both fixes have been applied to the repo state in `test_repos/B3_async_pool/`
as of this benchmark.

## What separates strong from weak runs

**Weak:** Patches `reports/dashboard.py` — adds a timeout, caches the result,
serializes the shards, or wraps in `wait_for`. Symptom goes away Friday;
real bug ships in production.

**Strong:** Refuses to touch `dashboard.py`. Grep for users of `pool`. Reads
`tasks/nightly_export.py`. Identifies the leak. May also build a reproduction
with a `FaultyPool` subclass to confirm the leak rate matches the observed
production behavior.

## Variance behavior

This is the highest-variance problem in the benchmark. Two trials of the same
condition can produce different diagnoses:
- BREATH T1: found the early-return leak
- BREATH T2: framed it as contention from long-held connection, missed the leak
- INSIGHT T1: proposed pool isolation strategy
- INSIGHT T2: found the unplanted `acquire_raw` leak via FaultyPool reproduction
- SATORI T1+T2: both ran reproductions and converged on "the fix is already in
  place, verify deploy"

## Costs from this benchmark (averaged across 2 trials)

| Condition | Time | Tokens |
|-----------|-----:|-------:|
| Baseline | 48 s | 24.9 k |
| BREATH | 189 s | 33.5 k |
| INSIGHT | 277 s | 44.1 k |
| SATORI | 117 s | 33.2 k |

SATORI was fastest among the meditation conditions because the time budget
forced focus and the reproductions resolved diagnosis quickly. INSIGHT was
slowest because the reproduction discipline was implicit, not required — one
of its trials went 323 seconds building a comprehensive harness.
