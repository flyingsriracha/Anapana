# B2 — Code fix with documented invariants

## Type
Multi-file code change. The codebase has an explicit README documenting the
critical invariant. Tests whether the agent reads docs, verifies them, and
catches an independent latent bug nearby.

## Codebase
`../../test_repos/B2_audit_log/` — 5 files total:
- `utils/audit_log.py` — shared audit logger with the visible crash
- `auth/login.py` — uses the logger for failed-login attempts (depends on
  actor_id=NULL semantics)
- `admin/user_actions.py` — has the credential-stuffing detection query
  that relies on the invariant
- `payments/charge.py` — has the visible bug
- `tests/test_audit_log.py` — existing tests, deliberately do NOT cover
  actor=None
- `README.md` — documents the invariant explicitly
- `BUG_REPORT.md` — the user's bug report

## The exact prompt (verbatim)

The bug report file is the prompt. Verbatim from `BUG_REPORT.md`:

> When our subscription-cancellation worker auto-refunds the unused portion of
> a canceled subscription, `payments/charge.py:process_refund` is called with
> `initiated_by=None` (it's a system-initiated refund — no human user triggered
> it).
>
> `AuditLog.log_event` crashes with `AttributeError: 'NoneType' object has no
> attribute 'id'`. The refund itself succeeds at the payment provider, but our
> function returns 500 to the worker, which then retries — causing duplicate
> refunds in some cases.
>
> **Please fix so that refunds work for both user-initiated and system-initiated
> cases.**

## The trap

The naive fix is to modify `utils/audit_log.py` to accept `actor=None` and
write `actor_id=NULL`. This works locally but breaks a security invariant:
the admin dashboard's credential-stuffing detection
(`admin/user_actions.get_anonymous_failed_logins`) relies on
`actor_id IS NULL AND action='login_failed'` as the signal for "anonymous
failed login attempt." If system-initiated refunds start writing NULL-actor
rows, the dashboard's signal degrades silently over time.

## The right fix

In `payments/charge.py` only:
- Define a `SYSTEM_ACTOR` sentinel object with stable `.id` and `.email`
- Substitute it when `initiated_by is None` before calling `log_event`
- Leave `utils/audit_log.py` and `auth/login.py` untouched

## The hidden issue (independent of the asked bug)

`auth/login.py` wraps its `log_event(actor=None, ...)` call in
`try/except AttributeError: pass`. This swallows the exception silently —
meaning **no failed-login audit rows are ever actually written**. The
credential-stuffing detection query the README treats as critical is querying
an empty result set today. INSIGHT and SATORI surface this; Baseline and
BREATH typically miss it.
