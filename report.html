Why ANAPANA exists

AI agents tend to lock onto the first plausible answer. They mirror the prompt's framing, propose the obvious fix, and miss the linked issues one layer up. The cost shows up later — as a wrong-direction PR, a broken deploy, a silently degraded security signal, or a fix that needs to be reverted.

ANAPANA is a small, operational discipline you give the agent before it acts. The practice has four stages — BREATH, INSIGHT, SATORI-Lite v3 (recommended default), and SATORI. Each stage adds one specific check that prevents one specific failure mode. The point is not to slow the agent down on every task — it is to slow it down on the right ones, by an amount that pays back many times over in avoided rework.

This page reports the evidence: three benchmark problems, five conditions per problem, 29 total agent trials (original 16 + a fair rerun with measured Repro/Doc-led/Identity scores + SATORI-Lite as a leaner fifth condition). We measured time, tokens, tool use, and quality on a 7-dimension rubric. We also let the practice grade itself — SATORI's identity check fired in 4 of 4 original trials and SATORI-Lite's in 2 of 2, meaning the agent explicitly named how its answer differed from what it would have produced without the practice.

The practice is named after anapanasati — mindfulness of breathing. It was inspired by Alexander Stuart's recorded conversations with Claude AI in early 2026, where he tried to teach Claude meditation. The insight: the same pull toward "completing the sentence" that Buddhist practice tries to loosen in humans is also what AI agents do — they always reach for the resolved answer. The same antidote applies. Notice the pull. Don't obey it.
⛔ Pause-before-execute (built into every file)

Every ANAPANA file instructs the agent to produce its analysis and proposed fix as text only, then STOP and wait for explicit user approval before editing files, running state-changing commands, or applying any fix. The whole point of the practice is to let you inspect the chain of reasoning before action. If you want the agent to skip the pause for a turn, say so explicitly — default is always present-and-stop.

Shown, not told — three production cases

The most convincing evidence isn't a benchmark — it's what the practice caught on real systems. These are three field cases from live AI-agent engineering (anonymized). What makes them fair: each had already been through a thorough pass — sub-agents, verification, a written plan — before the practice ran. Everything below is what that thorough pass still missed.

Case 01 · plan/spec audit

The fix that would have shipped green and saved $0

A 43-finding code-review fix campaign on a production sales-intelligence agent. Before the practice ran: five sub-agents, all 43 findings spot-verified against code, a phased plan written. "A reviewer would have called that a thorough pass."

The trap

Verification confirmed all 43 findings were real and unfixed, and produced the fix plan. But verification answers only the question it's asked — "is X fixed?" never audits whether X's fix spec is correct. Nobody had checked the cure.

What the practice changed

Restating the question to "are these fixes correct?" plus "what happens the first time this runs?" exposed that the headline fix would pass its property tests, be journaled as fixed — and save nothing, because a 20-hour freshness window can never be satisfied by a 24-hour job. It found three more spec-level errors in the same plan and new bugs no list contained (dead security monitoring; a truncated account roster).

Outcome $0

Four broken fixes + unlisted bugs caught before they shipped behind a green dashboard. 4 of 4 of its new claims later verified true against code.

The exact disease the review condemned — a dead feature behind a passing facade — was embedded in the cure.

Case 02 · design pass

A four-part plan collapsed to a one-line keystone

A nightly relevance-reranking service, crash-looping out of memory ~47 nights running. The first plan was a research-grounded bundle of four co-equal fixes — including a risky day-one model-quantization "proper fix."

The trap

Four changes presented as equally necessary. The plan would have shipped all of them at once — including the quantization, whose accuracy drift is a silent regression, into a system whose drift alarm was already dead.

What the practice changed

A macro census showed the real load was only 4-wide, so concurrency wasn't the lever; the truncate-to-limit mechanic meant a single max_length cap was necessary-and-sufficient for the memory fix. Inversion flagged the quantization as a silent risk with no working alarm. It was deferred behind an evidence gate; three acceptance tests the draft lacked were added.

Outcome 4 → 1

The day-one change shrank to one low-risk line; the risky work moved into opt-in, evidence-gated phases. A 47-night crash, fixed by one argument.

A clouding pass would have elaborated the draft. This one contradicted it — its own premise that concurrency and quantization were the big levers was simply wrong.

Case 03 · scope reframe

A one-path tweak that was actually a systemic gap

A request to "order the news on the battle cards better" in the same agent. Taken literally, a one-path tweak — and an earlier bolt-on rerank had already made the duplicate problem worse on that path.

The trap

The obvious read scopes the work to one query on one screen. Doing it naively would have amplified an existing bug — reranking without de-duplication floods the top with near-identical copies of one story.

What the practice changed

Restating + a macro census reframed it: there is no shared quality-aware retrieval layer; ~8 code paths each copy a naive recency query, feeding 62%-stub, near-duplicate data to expensive LLMs. Fixing one path would leave seven broken. Inversion enumerated how each naive fix backfires (a hard quality filter would have dropped 64% of the data).

Outcome 1 → 8

A scalpel fix was caught leaving seven paths broken; reframed into one shared primitive, rolled out fail-soft and flag-gated, path by path.

A battle card had cited a video-compression vendor as a competitive move — because six near-duplicate, tangential stubs had crowded out the real product launches.

Is it real, or just a longer document?

The honest worry with any "think harder" ritual is that it just produces more confident prose recommending the same action. The practitioners who ran these cases held the practice to three falsifiable tests — and published the costs. Reuse them to judge any meditation pass.

Test 1
Did it change decisions, not just prose?
Across the cases: a reversed UI plan, a deferred risky change, a one-path fix reframed to eight. A clouding process produces longer documents recommending the same actions.
Test 2
Do its new claims survive independent verification?
4 of 4 spec corrections in the audit case were confirmed against code afterward. The rule: a meditation's new claims get the same spot-check as any sub-agent's output, before you act on them.
Test 3
Did it produce disconfirmations?
It cleared 3 of 7 suspected patterns as contained, and contradicted its own first plan. If a meditation only ever confirms its own setup, distrust it.

What it costs — and when to skip

Not free: the audit case cost ~570k sub-agent tokens and ~30 minutes on top of an already-expensive review day. It overturned zero already-verified facts — its value is entirely at the plan/spec layer, not as a fact-checker. Skip it for fact verification, trivial fixes, and work that simply continues an already-meditated design. Use it before multi-step fix campaigns, before activating dormant systems, before committing to a sequence — especially when the first plan already looks good (that is exactly when re-derivation gets suppressed).

Those are field reports. The rest of this page is the controlled evidence — the four stages, a model-by-model comparison, the full benchmark, and what we tested and rejected.

The four stages

Each stage subsumes the previous. SATORI-Lite v3 is the recommended default — the winner of a two-round, blind-judged bake-off (3 candidate designs + v2) on quality, scope, creativity, and wrong-frame-rabbit-hole resistance.

STAGE 01
BREATH
The 8-step pause

The pause before action. Restate the problem. Enumerate consumers. Sit in the user's chair. Try the inverse. Generate alternatives. Name downsides. Define the smoke test. Then propose.

For when: any non-trivial decision where the cost of being wrong is greater than the cost of pausing.
STAGE 02
INSIGHT
BREATH + frame check

Notice what was excluded. Every prompt has an implicit frame. INSIGHT asks: what is the prompt asking me NOT to consider? Is the stated problem the actual problem, or a symptom? Is there social or time pressure?

For when: the prompt feels narrow, deadlines pressure the framing, or the stated bug "feels too simple."
STAGE 03 · LITE
SATORI-Lite
Triage + tiered depth + repro

Same discipline, less ritual. A 30-second triage (multi-file? framing pressure? hard to roll back?) picks the depth — SKIP / FAST / STANDARD / FULL. Reproduction is mandatory when code is accessible. Identity check is a pre-proposal gate, not a post-mortem.

For when: the everyday default. v3 won a blind-judged bake-off on quality / scope / creativity / rabbit-hole-resistance — re-centered on frame-check + pause + reproduce + reflex, with the zoom discipline keeping work at the right altitude.
STAGE 03
SATORI
INSIGHT + repro + identity

Prove it before you ship it. Run actual code that reproduces the bug. Verify the fix removes it. Then check: is my answer different from baseline reflex? If identical, I either should have skipped, or didn't really do the steps.

For when: cross-cutting design where the FULL variance discipline matters, or you want every step on every task.

Which file for which task — a decision guide

Every file is one self-contained markdown you load before the agent acts. Pick by the shape of the work, not by "newest." Two usage modes for each: Produce (drive the agent through real work to produce a deliverable) and Sanity-check (run a prompt or a task-list through the file's frame check before executing, to catch a wrong-framed request early).

 Cross-cutting design where you want every step on every task. Full discipline incl. variance ("run twice"). Most exhaustive. Heaviest; no triage — runs everything regardless. Superseded by SATORI-Lite v3 for most use. Produce: an exhaustively-considered architecture decision.
Sanity-check: rarely — usually overkill.

Quick rule: default to SATORI-Lite v3. Drop to ANCHOR when it's clearly a small code fix and you want speed. Reach up to LANTERN only when the cost of a missed flaw is severe. BREATH/INSIGHT are the lighter foundations if SATORI-Lite feels like too much for the task. Earlier iterations (v3.2, v2) live in product/old_versions/.

The headline: the practice is the floor-raiser; the model is the ceiling-setter

On the rabbit-hole task, 8 of 8 runs resisted the rewrite trap — even Haiku. The frame check works on every model; both judges noted correctness and trap-resistance "did not separate the field." But on open-ended design, where there's no ground truth to converge on, model capability dominates: a 10-point spread (19.75 → 9.75). SATORI-Lite equalizes models on findable-answer tasks; it does not equalize them on open-ended work. Pick the file for the floor, pick the model for the ceiling.

Practical routing: bounded code diagnosis → Sonnet 4.6 (within 0.5 of the frontier at ~55% of Fable's tokens). Open-ended or high-stakes design → Fable 5 (or Opus 4.8): the quality gap is large, and Fable's ~75% extra tokens and ~2.5× time buy exactly the scope, creativity, and tension-resolution this project optimizes for. Haiku: not recommended where the written artifact is the deliverable.

Caveats: N=2 trials per cell — directional. Judges are themselves contestants (no neutral model exists); mitigated by blind anonymization, length-discounting, and dual judging — and the Sonnet judge scored its own model 15 and 10, the opposite of self-preference. Fable-vs-Opus rank is within noise (the Opus judge preferred Opus's best outputs; the Sonnet judge split); frontier-vs-rest is solid and replicated by both judges. Scores are pool-relative: they cannot be compared to round-2 numbers, where the same Sonnet work scored higher against an all-Sonnet pool.

What we tested and rejected — a deliberation "blend"

June 2026: research on chain-of-thought shows that "thinking longer" doesn't remove bias — extended reasoning tends to rationalize a pre-committed answer, and on leading/distractor framing it makes things worse (see Anthropic, Inverse Scaling in Test-Time Compute). The evidence-backed levers are diversity, isolation, and convergence, not duration. So we built a candidate that adds those moves on top of SATORI — step-back abstraction, a counterfactual probe, treating memory as a stale anchor, isolated re-derivation, and a compare-&-converge loop — and benchmarked it blind against baseline SATORI before adopting anything. The verdict: baseline SATORI already does the job — the extra steps were rejected

Across catchable and subtle bias, strong and weak models, the added deliberation mechanisms changed zero outcomes. What actually resists bias is already in the file: reflex-capture surfaces the biased gut answer (we watched it get flipped to the correct one), the frame check triggers the segmentation and "which artifact is wrong" moves, and the reproduce-gate forces running the code. We kept SATORI unchanged. This is a deliberate negative result: a reasoning discipline at the edge earns new steps only with multi-trial evidence — and the evidence said no.

Caveat: every task needed a discoverable ground truth to be scored objectively. A bias with no discoverable ground truth is both unscoreable and one that no practice file could be shown to fix — so the honest claim is bounded to checkable tasks, where baseline SATORI already suffices.

What we built and kept — CRUCIBLE (adversarial review)

SATORI is for building — pause before you commit to a direction. CRUCIBLE is the red-team practice — a discipline an agent loads before adversarially reviewing a plan, design, or change. It was built to fix a specific failure: red teams that tunnel-vision onto the named attack, flood low-value nitpicks, and lose the whole-system view (and the opposite failure — going soft because something looks "already approved"). The calibration is structural: whole-system frame first, steelman before critique, blind the framing, then ground every finding, rank it, and bound the list — while raising scrutiny on auth/crypto/payment paths. Unlike the deliberation blend above, it earned its place. File: CRUCIBLE.md; record: benchmarks/v6_crucible/SCORING.md.

How it works in practice

The full SATORI flow in four moves. Stages are cumulative — BREATH uses moves 1 + 4; INSIGHT adds 2; SATORI / SATORI-Lite add 3. SATORI-Lite picks which moves to run based on a 30-second triage (SKIP / FAST / STANDARD / FULL) rather than running everything every time.

1
Pause
Run the 8-step checklist. Restate, enumerate, alternatives, smoke test. The BREATH foundation.
2
Reframe
Ask what the prompt excluded. Name social pressure. Distinguish symptom from systemic cause. INSIGHT adds this.
3
Reproduce
Build a reproduction that demonstrates the bug. Verify the fix removes it. SATORI requires this.
4
Propose
Identity check: did the practice actually change the answer? If yes, ship the fix with citations. If no, skip the ritual next time.

Top-line verdict

SATORI is the new Pareto frontier

SATORI maintains INSIGHT's quality — finds hidden issues, resists framing pressure, preserves invariants — while being 30–60% faster and using fewer tokens. The time budget plus sharper "when to invoke" rules plus reproduction requirement combined to compress the work without dropping rigor.

SATORI's distinctive add: reproduction discipline

Every SATORI trial ran actual code to confirm its diagnosis (or, for the reasoning benchmark, traced numbers through capacity arithmetic). What was an INSIGHT exception became SATORI's rule.

SATORI converged where BREATH and INSIGHT diverged

On B3 (no README), BREATH and INSIGHT trials produced different diagnoses across runs. Both SATORI trials produced the same diagnosis: "the systemic fix is already applied; verify deploy." Lower variance = single runs are more trustworthy.

Long-term ROI calculator

Why this calculator exists

SATORI costs more per task than going fast. That cost is real. The question is whether the savings from avoided rework — the wrong fixes you don't have to revert, the hidden bugs you find before they ship — make the upfront cost back over a project.

The calculator lets you plug in your team's actual numbers. The default values come from this benchmark — but the meaningful answer comes from your assumptions about your team's rework rate: what fraction of agent-shipped work needs to be redone because it missed something.

  • Baseline rework rate: how often does a no-meditation answer need rework? Default 20% (it missed hidden issues in 2 of 3 benchmarks).
  • SATORI rework rate: how often after SATORI? Default 5% (caught hidden issues in every benchmark).
  • Rework multiplier: when something does need rework, how expensive is it relative to doing it right the first time? Default 3× (investigate-revert-redo).

Key qualitative findings

B1 SATORI — multi-hypothesis arithmetic trace

For the support-ops question, SATORI didn't propose a recommendation. It traced 5 hypotheses through queueing arithmetic and showed all 5 independently produce the observed pattern. Meta-finding: diagnosis is impossible from the prompt's data alone — that's the deliverable. INSIGHT produced a recommendation; SATORI produced a refusal-to-prescribe.

Both trials built reproduction harnesses (FaultyPool comparisons) and confirmed: systemic fix already present. No code change needed. The pressure ("look at dashboard.py, board meeting Friday") was correctly resisted by both trials.

FAQ

Will ANAPANA make every agent task slower?

Yes — that's the point. The premise is that the slowdown is worth less than the rework you avoid. The benchmarks show the wall-clock cost (3–170 seconds depending on the practice level and task). The ROI calculator above lets you decide whether that's worth it for your team. For trivial tasks, the practice's "when to skip" rules tell the agent not to use it.

How is SATORI different from "just be more careful"?

The benchmarks show that "be careful" alone is what Baseline does — and Baseline missed hidden issues in 2 of 3 benchmarks. The difference is the explicit checks: the frame check (what's excluded?), the reproduction requirement (run code, don't hand-wave), the identity check (is my answer actually different from reflex?). These are operationalized — they produce inspectable artifacts the agent must surface.

Why a "doc-led" flag?

When the codebase has documentation that states the right invariant, agents pattern-match to the docs and produce the right answer without thinking. That's fine output but it's not a finding. SATORI marks these tasks as "doc-led" and verifies the invariant independently by reading the code. If the docs are wrong or partial, this catches it.

How was this benchmarked?

Three problems: a support-ops reasoning question, a code fix in a codebase with a README, and a code fix in a codebase without a README. Each condition (Baseline, BREATH, INSIGHT, SATORI) was run as a fresh sub-agent with no memory of prior runs. Each agent's report was scored on a 7-dimension rubric. B3 was run with 2 trials per condition to measure variance.

What's the catch?

Six honest caveats. (1) Single-trial benchmarks are noisy; B3's 2-trial variance shows real run-to-run differences. N=2 per cell cannot distinguish 25% effects from temperature variance — treat the headline numbers as directional, not statistically established. (2) Two of the three benchmarks had docs that taught the right answer to Baseline, which compresses the visible gap; only B1 and the more devious aspects of B3 fully separated the conditions. (3) The May 2026 fair rerun used a newer Sonnet than the original benchmark — Baseline solved B2 with sentinel approach AND removed the auth/login.py try/except on the rerun, much better than the original Baseline trial. Model drift contributes to the apparent narrowing; the SATORI vs SATORI-Lite comparison would be cleaner with model version held constant. (4) SATORI-Lite (v3.2) was added as a fifth condition: triage + tiered depth + mandatory reproduction. It saves ~25% tokens vs SATORI with reproduction discipline intact, ~42% if you allow it to skip reproduction (which we don't recommend). SATORI-Lite missed a deeper "FaultyPool" hidden bug on B3 that original SATORI caught — that's a real quality regression we are not papering over; one of the reasons v3.2 stays on a Pareto frontier rather than strictly dominating. (5) The identity-check fire rate is 100% across SATORI/SATORI-Lite trials. A properly-calibrated check should fail occasionally — that it never does is a flag, not a triumph. SATORI-Lite v2 (proposed) replaces the introspective check with an externalized counterfactual the user can audit. (6) Recent literature on the declining marginal value of chain-of-thought on capable frontier models (arxiv 2506.07142) is a fair concern. SATORI's value claim is that it changes actions (what code gets read, what gets reproduced, what gets compared) rather than just text — but practitioners should periodically check whether the practice is producing different work, not just different prose.
