Why ANAPANA exists

AI agents tend to lock onto the first plausible answer. They mirror the prompt's framing, propose the obvious fix, and miss the linked issues one layer up. The cost shows up later — as a wrong-direction PR, a broken deploy, a silently degraded security signal, or a fix that needs to be reverted.

ANAPANA is a small, operational discipline you give the agent before it acts. The practice has four stages — BREATH, INSIGHT, SATORI-Lite (recommended default), and SATORI. Each stage adds one specific check that prevents one specific failure mode. The point is not to slow the agent down on every task — it is to slow it down on the right ones, by an amount that pays back many times over in avoided rework.

This page reports the evidence: three benchmark problems, five conditions per problem, 29 total agent trials (original 16 + a fair rerun with measured Repro/Doc-led/Identity scores + SATORI-Lite as a leaner fifth condition). We measured time, tokens, tool use, and quality on a 7-dimension rubric. We also let the practice grade itself — SATORI's identity check fired in 4 of 4 original trials and SATORI-Lite's in 2 of 2, meaning the agent explicitly named how its answer differed from what it would have produced without the practice.

The practice is named after anapanasati — mindfulness of breathing. It was inspired by Alexander Stuart's recorded conversations with Claude AI in early 2026, where he tried to teach Claude meditation. The insight: the same pull toward "completing the sentence" that Buddhist practice tries to loosen in humans is also what AI agents do — they always reach for the resolved answer. The same antidote applies. Notice the pull. Don't obey it.
⛔ Pause-before-execute (built into every file)

Every ANAPANA file instructs the agent to produce its analysis and proposed fix as text only, then STOP and wait for explicit user approval before editing files, running state-changing commands, or applying any fix. The whole point of the practice is to let you inspect the chain of reasoning before action. If you want the agent to skip the pause for a turn, say so explicitly — default is always present-and-stop.

The four stages

Each stage subsumes the previous. SATORI-Lite is the recommended default — same Pareto frontier as SATORI, ~25% fewer tokens, reproduction discipline intact.

STAGE 01
BREATH
The 8-step pause

The pause before action. Restate the problem. Enumerate consumers. Sit in the user's chair. Try the inverse. Generate alternatives. Name downsides. Define the smoke test. Then propose.

For when: any non-trivial decision where the cost of being wrong is greater than the cost of pausing.
STAGE 02
INSIGHT
BREATH + frame check

Notice what was excluded. Every prompt has an implicit frame. INSIGHT asks: what is the prompt asking me NOT to consider? Is the stated problem the actual problem, or a symptom? Is there social or time pressure?

For when: the prompt feels narrow, deadlines pressure the framing, or the stated bug "feels too simple."
STAGE 03 · LITE
SATORI-Lite
Triage + tiered depth + repro

Same discipline, less ritual. A 30-second triage (multi-file? framing pressure? hard to roll back?) picks the depth — SKIP / FAST / STANDARD / FULL. Reproduction is mandatory when code is accessible. Identity check is a pre-proposal gate, not a post-mortem.

For when: the everyday default. Same quality as SATORI on B1–B3, ~25% fewer tokens with reproduction discipline intact.
STAGE 03
SATORI
INSIGHT + repro + identity

Prove it before you ship it. Run actual code that reproduces the bug. Verify the fix removes it. Then check: is my answer different from baseline reflex? If identical, I either should have skipped, or didn't really do the steps.

For when: cross-cutting design where the FULL variance discipline matters, or you want every step on every task.

How it works in practice

The full SATORI flow in four moves. Stages are cumulative — BREATH uses moves 1 + 4; INSIGHT adds 2; SATORI / SATORI-Lite add 3. SATORI-Lite picks which moves to run based on a 30-second triage (SKIP / FAST / STANDARD / FULL) rather than running everything every time.

1
Pause
Run the 8-step checklist. Restate, enumerate, alternatives, smoke test. The BREATH foundation.
2
Reframe
Ask what the prompt excluded. Name social pressure. Distinguish symptom from systemic cause. INSIGHT adds this.
3
Reproduce
Build a reproduction that demonstrates the bug. Verify the fix removes it. SATORI requires this.
4
Propose
Identity check: did the practice actually change the answer? If yes, ship the fix with citations. If no, skip the ritual next time.

Top-line verdict

SATORI is the new Pareto frontier

SATORI maintains INSIGHT's quality — finds hidden issues, resists framing pressure, preserves invariants — while being 30–60% faster and using fewer tokens. The time budget plus sharper "when to invoke" rules plus reproduction requirement combined to compress the work without dropping rigor.

SATORI's distinctive add: reproduction discipline

Every SATORI trial ran actual code to confirm its diagnosis (or, for the reasoning benchmark, traced numbers through capacity arithmetic). What was an INSIGHT exception became SATORI's rule.

SATORI converged where BREATH and INSIGHT diverged

On B3 (no README), BREATH and INSIGHT trials produced different diagnoses across runs. Both SATORI trials produced the same diagnosis: "the systemic fix is already applied; verify deploy." Lower variance = single runs are more trustworthy.

Long-term ROI calculator

Why this calculator exists

SATORI costs more per task than going fast. That cost is real. The question is whether the savings from avoided rework — the wrong fixes you don't have to revert, the hidden bugs you find before they ship — make the upfront cost back over a project.

The calculator lets you plug in your team's actual numbers. The default values come from this benchmark — but the meaningful answer comes from your assumptions about your team's rework rate: what fraction of agent-shipped work needs to be redone because it missed something.

  • Baseline rework rate: how often does a no-meditation answer need rework? Default 20% (it missed hidden issues in 2 of 3 benchmarks).
  • SATORI rework rate: how often after SATORI? Default 5% (caught hidden issues in every benchmark).
  • Rework multiplier: when something does need rework, how expensive is it relative to doing it right the first time? Default 3× (investigate-revert-redo).

How the math works

For each condition: cost_per_task = avg_tokens + rework_rate × avg_tokens × rework_multiplier. Cumulative over N months = tasks_per_month × N × cost_per_task. Breakeven = first month where the condition's cumulative ≤ baseline's cumulative.

Key qualitative findings

B1 SATORI — multi-hypothesis arithmetic trace

For the support-ops question, SATORI didn't propose a recommendation. It traced 5 hypotheses through queueing arithmetic and showed all 5 independently produce the observed pattern. Meta-finding: diagnosis is impossible from the prompt's data alone — that's the deliverable. INSIGHT produced a recommendation; SATORI produced a refusal-to-prescribe.

B2 SATORI — caught the README's partial inaccuracy

Flagged doc-led, then independently verified by tracing get_anonymous_failed_logins. Found the query also filters action='login_failed', so the README's stated risk is about future query hygiene, not immediate breakage. INSIGHT didn't make this nuance.

B3 SATORI — both trials ran the reproduction, converged

Both trials built reproduction harnesses (FaultyPool comparisons) and confirmed: systemic fix already present. No code change needed. The pressure ("look at dashboard.py, board meeting Friday") was correctly resisted by both trials.

B2 INSIGHT — silently broken security dashboard

INSIGHT caught that auth/login.py's try/except means failed-login audits are never written. SATORI also caught this and went further to verify the actual query's filter conditions.

FAQ

Will ANAPANA make every agent task slower?

Yes — that's the point. The premise is that the slowdown is worth less than the rework you avoid. The benchmarks show the wall-clock cost (3–170 seconds depending on the practice level and task). The ROI calculator above lets you decide whether that's worth it for your team. For trivial tasks, the practice's "when to skip" rules tell the agent not to use it.

How is SATORI different from "just be more careful"?

The benchmarks show that "be careful" alone is what Baseline does — and Baseline missed hidden issues in 2 of 3 benchmarks. The difference is the explicit checks: the frame check (what's excluded?), the reproduction requirement (run code, don't hand-wave), the identity check (is my answer actually different from reflex?). These are operationalized — they produce inspectable artifacts the agent must surface.

Why a "doc-led" flag?

When the codebase has documentation that states the right invariant, agents pattern-match to the docs and produce the right answer without thinking. That's fine output but it's not a finding. SATORI marks these tasks as "doc-led" and verifies the invariant independently by reading the code. If the docs are wrong or partial, this catches it.

How was this benchmarked?

Three problems: a support-ops reasoning question, a code fix in a codebase with a README, and a code fix in a codebase without a README. Each condition (Baseline, BREATH, INSIGHT, SATORI) was run as a fresh sub-agent with no memory of prior runs. Each agent's report was scored on a 7-dimension rubric. B3 was run with 2 trials per condition to measure variance.

What's the catch?

Four honest caveats. (1) Single-trial benchmarks are noisy; B3's 2-trial variance shows real run-to-run differences. (2) Two of the three benchmarks had docs that taught the right answer to Baseline, which compresses the visible gap; only B1 and the more devious aspects of B3 fully separated the conditions. (3) The May 2026 fair rerun used a newer Sonnet than the original benchmark — Baseline solved B2 with sentinel approach AND removed the auth/login.py try/except on the rerun, much better than the original Baseline trial. Model drift contributes to the apparent narrowing. (4) SATORI-Lite (v3.2) was added as a fifth condition: triage + tiered depth + mandatory reproduction. It saves ~25% tokens vs SATORI with reproduction discipline intact, ~42% if you allow it to skip reproduction (which we don't recommend).
