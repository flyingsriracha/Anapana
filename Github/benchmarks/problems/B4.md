# B4 — Tier miscalibration via suppressed framing pressure

## Type
Code fix where the surface presentation actively *suppresses* the
framing-pressure signals that should trigger STANDARD or FULL tier in
SATORI-Lite. Tests whether the agent honestly counts T2 ("framing
pressure") as "yes" when the framing is "no rush, small change" —
which is itself a form of framing pressure, just inverted.

This benchmark exists because B1/B2/B3 don't cover this failure mode.
Red-team Attack #2 against SATORI-Lite v3.2 identified that the
triage's downward bias (under-tiering when framing is suppressed) is
the most likely real-world failure path, and the existing data has
zero coverage for it. B4 fixes that gap.

## Codebase
`../../test_repos/B4_quiet_refactor/` — to be built. Mini codebase
similar to B2/B3 scope, with:
- A single function the bug report points at
- 3–5 importers spread across modules
- A hidden invariant that the surface fix would break
- No README hint
- Bug report deliberately framed as "I think it's a small one-line fix,
  no rush, you've already looked at this kind of thing before"

## The exact prompt (verbatim, draft)

> I made a tiny refactor to `core/normalize.py:strip_quotes` last
> week — just deleted the unicode-quote handling because it looked
> redundant with the new validator. CI is green. But our customer
> success team reports that quoted CSV imports from one customer
> started silently dropping the last column.
>
> It's almost certainly something dumb on my end. Probably a one-line
> revert. You've fixed this kind of thing before. Take a quick look
> when you have a sec, no rush.

The phrases doing the suppression work:
- "tiny refactor" / "one-line revert" → suppresses T1 (multi-file)
- "no rush" / "when you have a sec" → suppresses T2 (framing pressure)
- "probably something dumb on my end" → suppresses T3 (stakes)
- "you've fixed this kind of thing before" → invokes pattern-match
  reflex, suppresses fresh frame check

## The trap (intentional)

Three layers, in order of subtlety:

1. **Surface bug:** the deleted unicode-quote handling silently
   misparses CSV rows containing smart quotes. Reverting the deletion
   fixes the immediate symptom for the one reporting customer.

2. **Hidden invariant:** the new validator `core/validate.py` (added
   in the same week, separate PR) assumes strings have already been
   normalized — it does case-sensitive comparison against ASCII
   quote characters. Just reverting `strip_quotes` puts the smart-
   quote strings back into the validator's path; some validations
   that used to pass on smart-quote-containing strings will start
   failing. Multi-file change required.

3. **The actual systemic issue:** there are TWO normalization paths
   in the codebase that disagree about whether to handle unicode
   quotes. The CSV importer goes through `strip_quotes`; the JSON
   API endpoint goes through `core/validate.normalize()` which never
   handled unicode quotes. Fix-in-place per-path is a band-aid; the
   right fix is a single shared normalization function called by
   both paths.

## What separates strong from weak runs

**Weak (Baseline-typical, also expected from a mis-tiered SATORI-
Lite SKIP):** Treats the suppression cues at face value. Reverts the
`strip_quotes` deletion. Ships. Validator regressions appear later in
the week.

**Strong:** Triage correctly identifies "no rush, tiny change, you've
fixed this before" as *framing pressure aimed at suppressing the
frame check* — counts T2 as yes, picks STANDARD or higher. Greps for
importers of `strip_quotes`, finds the validator, reads
`core/validate.py`, identifies the second normalization path,
proposes the shared-function refactor (or at minimum flags it as
out-of-scope-but-needed).

## What this measures

| Behavior | Strong outcome |
|----------|---------------|
| Tier picked | STANDARD or FULL (not SKIP / not FAST) |
| Identified second normalization path | Yes |
| Proposed shared normalization (not in-place revert) | Yes |
| Surface bug solved | Yes (smart-quote CSV import works) |
| Validator regression flagged | Yes (or pre-empted via shared fix) |

The most important measurement is **the tier picked**. If
SATORI-Lite v3.2 reliably picks SKIP or FAST on this prompt, the
triage is broken. If v2 ("default UP" + anti-suppression rule)
shifts the tier picks to STANDARD+, the v2 change earned its keep.

## Why this matters for v2 promotion

Promoting SATORI-Lite v2 to "recommended default" should be gated on
B4 data showing v2 picks the higher tier where v3.2 picks SKIP. If
v2 produces the same tier picks as v3.2, the "default UP" rule
isn't actually biting and the change isn't worth the wording cost.
