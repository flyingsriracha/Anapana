6. **Alternatives and downsides:**
   - *Option A — Sentinel system-user object:* Caller passes a stable `SYSTEM` sentinel (e.g., `SystemActor(id="system", email="system@internal")`). `log_event` unchanged. Downside: every existing caller that passes `None` (charge.py, login.py workaround) must be updated; breaks if callers forget. Silent failure mode: forgotten callers still crash.

Downside: system-actor refund rows get `actor_id=NULL`, polluting `get_anonymous_failed_logins` query and diluting credential-stuffing signal as documented in README and admin/user_actions.py. Silent failure mode: security dashboard silently degrades — no error, no alert.
   - *Option C — Two distinct branches in `log_event` (preferred):* Accept `actor=None` but write a sentinel string (e.g., `actor_id="system"`, `actor_email=None`) instead of `NULL`, so rows are still queryable and do NOT satisfy `actor_id IS NULL`. Downside: `actor_id` column becomes mixed type (int for real users, string `"system"` for daemon). ## PROPOSED_FIX
**File: `utils/audit_log.py`** — modify `log_event` to handle `actor=None` by writing a stable string sentinel `"system"` rather than `NULL`, preserving the `actor_id IS NULL` invariant used by `get_anonymous_failed_logins`.

```python
def log_event(self, actor, action, target, metadata=None):
    if actor is None:
        actor_id = "system"
        actor_email = None
    else:
        actor_id = actor.id
        actor_email = actor.email

    record = {
        "actor_id": actor_id,
        "actor_email": actor_email,
        "action": action,
        "target_id": target.id,
        "metadata": metadata or {},
        "timestamp": datetime.utcnow(),
    }
    self.db.insert("audit_log", record)
```

**File: `auth/login.py`** — remove the silent try/except workaround (lines 28-35). Now that `log_event` properly supports `actor=None`, the swallowed exception should be unwrapped so failed-login audit events are no longer silently dropped:

```python
audit.log_event(
    actor=None,
    action="login_failed",
    target=_FailedLoginTarget(email),
)
```

**Note on sentinel value:** Using `"system"` (string) for `actor_id` keeps it non-NULL, preserving the security query invariant. Teams should agree on a stable sentinel convention (e.g., `"system"` or `-1`) and document it. Option D (a real DB row) is architecturally cleaner for the long term.

## HIDDEN_ISSUES
1. **Silent audit data loss in `auth/login.py`:** The existing `try/except AttributeError: pass` (lines 28-35) has been silently dropping every failed-login audit event since it was added. The credential-stuffing dashboard has therefore been running on incomplete data. This is a security-relevant data-loss bug that predates the refund bug and persists regardless of which fix is chosen.

2. **`actor_id` type pollution risk:** If the proposed sentinel-string fix is applied, the column becomes mixed-type (int vs. string). Downstream queries using `actor_id = ?` with an integer comparison may silently miss system rows. Should be documented and, ideally, schema-constrained.

3. **No test coverage for `actor=None`:** `tests/test_audit_log.py` line 47-49 explicitly notes this gap. Any fix needs a new test.
